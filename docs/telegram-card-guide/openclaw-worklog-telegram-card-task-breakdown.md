# OpenClaw Worklog Telegram 卡片化交互开发任务拆分

## 1. 开发目标

交付 `openclaw-worklog` 的 Telegram 卡片化一期能力：

- 主菜单卡片
- 记录日志输入态
- 今日记录卡片
- 本月统计卡片
- 文本命令共存

## 2. 按模块拆分

### 模块 A：入口与主菜单

目标：让用户能稳定进入工作日志卡片首页。

任务：

- [ ] 明确入口口径：现有命令 / 新增别名分别是什么
- [ ] 增加主菜单消息渲染函数
- [ ] 增加主菜单 Inline Keyboard 构建函数
- [ ] 增加统一“返回主菜单”动作

验收：

- [ ] `工作日志` 可进入主菜单
- [ ] 若实现 `/worklog`，它也可进入主菜单
- [ ] 任意一期子页面都能回到主菜单

### 模块 B：callback 路由层

目标：建立最小且稳定的 callback 协议。

任务：

- [ ] 定义紧凑的 callback 前缀与枚举值
- [ ] 增加 Telegram callback handler
- [ ] 为未知 / 过期 callback 提供统一兜底
- [ ] 统一按钮动作到同一套路由入口

验收：

- [ ] 已定义按钮都能正确进入对应流程
- [ ] 未知 callback 不会抛异常

### 模块 C：输入态状态机

目标：支持按钮进入输入态，而不是只能靠纯文本命令。

任务：

- [ ] 在状态存储中补充 Telegram 输入态字段
- [ ] 增加 `awaiting_worklog_input`
- [ ] 增加 `awaiting_hours_only`
- [ ] 增加取消 / 返回 / 超时处理
- [ ] 明确文本命令对输入态的打断优先级

验收：

- [ ] 状态按 `channel + sender_id` 隔离
- [ ] 状态超时不会误吃后续普通消息
- [ ] 文本命令可打断输入态

### 模块 D：记录日志主链路

目标：打通“按钮进入输入态 → 用户输入 → 成功写入”。

任务：

- [ ] 输出记录日志卡片或提示消息
- [ ] 解析标准输入格式
- [ ] 复用现有写入能力完成日志追加
- [ ] 输出成功反馈
- [ ] 输出输入错误提示

验收：

- [ ] 点击 `➕ 记录日志` 可进入输入态
- [ ] 标准输入可成功写入
- [ ] 写入失败时有简短纠错提示

### 模块 E：今日记录卡片

目标：把“今日摘要”做成卡片入口。

任务：

- [ ] 汇总今日条数与工时
- [ ] 抽取最近一条摘要
- [ ] 处理空状态
- [ ] 提供“再记一条”“返回”动作

验收：

- [ ] 点击 `📋 今日记录` 可返回摘要
- [ ] 空状态不报错
- [ ] 可从此页继续进入记录流程

### 模块 F：本月统计卡片

目标：把现有统计口径包装成卡片展示。

任务：

- [ ] 汇总本月累计工时
- [ ] 计算应完成工时 / 未完成工时
- [ ] 输出简洁卡片文案
- [ ] 保留返回主菜单动作

验收：

- [ ] 点击 `📊 本月统计` 可返回统计结果
- [ ] 数据口径不偏离现有逻辑

### 模块 G：文本命令共存

目标：卡片化不打烂现有老入口。

任务：

- [ ] 梳理现有关键词与命令口径
- [ ] 保证按钮流程与文本流程共用后端逻辑
- [ ] 处理输入态与文本命令优先级

验收：

- [ ] 现有高频文本命令继续可用
- [ ] 卡片入口不会引入命令回归

## 3. 当前不建议纳入一期的任务

以下任务先明确排除，避免范围失控：

- [ ] 日志本重命名 UI
- [ ] 新增日志本 UI
- [ ] 切换日志本 UI
- [ ] 编辑最近一条日志
- [ ] 删除日志
- [ ] 列表分页 / 明细页

原因：

- 这些能力需要额外 service 支撑，已经超出“一期只做交互增强”的范围

## 4. 关键设计约束

### 4.1 不重复造轮子

- 按钮层不复制现有 append / access / guard / storage 逻辑
- 卡片只是 UI 入口，不是第二套 worklog 系统

### 4.2 状态必须隔离

- 至少按 `channel + sender_id` 隔离
- 若群聊存在 thread/topic，还应保留扩展位

### 4.3 callback 必须短

- 不在 callback 中塞长文本
- 不默认塞长 ID / 路径 / 原始工作项内容
- 如需动态对象，改用短 token

### 4.4 文案必须短

- 不写说明书式长文
- 一条消息只表达一个操作意图
- 默认短句、命令式、卡片式

## 5. 联调检查清单

### 主流程

- [ ] 主菜单可打开
- [ ] 记录日志可进入输入态
- [ ] 输入标准格式可成功写入
- [ ] 成功后有结果反馈

### 查询流程

- [ ] 今日记录可查看
- [ ] 本月统计可查看
- [ ] 空状态展示正常

### 共存流程

- [ ] 现有文本命令继续可用
- [ ] 输入态下文本命令可打断当前状态

### 异常流程

- [ ] callback 过期有兜底
- [ ] 输入格式错误有提示
- [ ] 状态超时后不会误处理普通消息

## 6. 提交建议

推荐按以下粒度拆 commit：

1. `feat(worklog): add telegram main menu and callback protocol`
2. `feat(worklog): add telegram input state flow for worklog append`
3. `feat(worklog): add today and monthly summary cards`
4. `refactor(worklog): unify telegram card flow with existing command logic`

## 7. 一句话结论

一期开发任务要盯死主链路，别被“日志本管理、编辑、删除”这些更贵的需求提前拖偏。
