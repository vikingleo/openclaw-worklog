# OpenClaw Worklog Telegram 卡片化交互分期计划

## 1. 分期目标

这份排期的核心不是“把所有想法都写进去”，而是保证：

- 一期能交付
- 一期能验收
- 一期不依赖大规模底层重构

## 2. 一期（MVP）

### 目标

交付 Telegram 卡片式最小可用版本。

### 范围

- 主菜单卡片
- 记录日志输入态
- 今日记录卡片
- 本月统计卡片
- 文本命令共存
- callback 过期 / 状态超时 / 返回 / 取消兜底

### 一期明确不做

- 日志本重命名
- 新增日志本
- 切换日志本卡片页
- 单条记录编辑
- 单条记录删除
- 明细分页
- 跨渠道 UI 复用

### 一期验收标准

- [ ] `工作日志` 可进入主菜单
- [ ] 如实现 `/worklog`，它可进入主菜单
- [ ] `➕ 记录日志` 可进入输入态
- [ ] 标准输入可成功写入日志
- [ ] `📋 今日记录` 可查看摘要
- [ ] `📊 本月统计` 可查看统计
- [ ] 原有文本命令不回归
- [ ] 页面支持返回 / 取消 / 超时兜底

### 一期价值

- 新用户能直接上手
- 老用户不被回归拖死
- 能尽快验证卡片式交互是否成立

## 3. 二期（增强）

### 目标

在一期稳定后，再补更复杂的交互管理能力。

### 范围

- 今日记录“查看更完整内容”
- 更顺手的统计回显
- 若已有稳定 service，再评估：
  - 日志本切换
  - 日志本重命名
  - 新增日志本

### 二期注意点

这些功能都带权限和状态复杂度，不应默认承诺“顺手就做”。

## 4. 三期（重功能）

### 目标

处理真正会碰到底层数据模型的问题。

### 范围

- 单条记录编辑
- 单条记录删除
- 记录级别短 token 定位机制
- 更复杂的列表 / 明细 / 分页

### 风险

这已经不是简单 UI 增强，而是会碰存储模型与安全边界的改造。

## 5. 推荐开发顺序

### 第一批

1. 主菜单卡片
2. callback 协议
3. 输入态状态机

### 第二批

4. 记录日志主链路
5. 成功 / 失败反馈

### 第三批

6. 今日记录卡片
7. 本月统计卡片

### 第四批

8. 文本命令兼容回归
9. 取消 / 返回 / 超时 / 过期兜底
10. 文案统一

## 6. 为什么这样分期

因为当前仓库现成稳定的能力，是：

- 追加日志
- 查询定位
- 权限检查
- 预览

而不是：

- 原地编辑单条记录
- 删除单条记录
- 动态日志本管理 UI

所以最合理的策略就是：

- 一期先贴着已存在能力做 UI 增强
- 更贵的能力留到二期、三期

## 7. 建议提交节奏

- `feat(worklog): add telegram main menu and callback protocol`
- `feat(worklog): add telegram input state flow for append`
- `feat(worklog): add today and monthly summary cards`
- `refactor(worklog): unify telegram card flow with command logic`

## 8. 最终结论

一期的目标不是“功能看起来很多”，而是“真正能上线验证”。

先把路修平：

- 能进
- 能记
- 能看
- 能查
- 不回归

做到这 5 条，再往后谈雕花，才不至于一边画饼一边返工。
