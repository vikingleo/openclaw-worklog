# OpenClaw Worklog Telegram 卡片化交互开发计划

## 1. 目标

将 `openclaw-worklog` 的 Telegram 交互从“文本命令优先”升级为“卡片按钮优先 + 文本命令兼容”。

但这次改造必须贴着现有仓库能力走，不能假设它已经是一个完整的 Telegram 机器人应用。

核心目标：

- 降低 Telegram 中记工作日志的使用门槛
- 保留现有文本命令能力，避免老用户回归
- 让按钮只负责交互入口与状态切换
- 让写入、查询、统计继续复用已有存储与权限逻辑
- 一期只交付最值钱的 4 个页面：主菜单、记录日志输入态、今日记录、本月统计

## 2. 当前基线

当前仓库已具备：

- 日志本解析与绑定
- 日志写入
- 月文件定位
- 锐评写入
- 读写权限控制
- 状态文件持久化
- 预览服务
- CLI 工具

当前仓库未明确具备：

- 完整 Telegram callback handler
- Telegram 输入态状态机
- 卡片消息构建层
- 单条记录编辑 / 删除能力
- 日志本重命名 / 创建的稳定 service

所以本计划必须分清：

- 哪些是“一期可直接接现有能力”
- 哪些是“二期以后再扩 service”

## 3. 一期范围

一期只做 4 个页面 / 流程：

1. 主菜单卡片
2. 记录日志输入态
3. 今日记录卡片
4. 本月统计卡片

并满足：

- 文本命令继续可用
- 按钮流程与文本命令共用后端能力
- 输入态可取消、可超时、可返回主菜单

## 4. 一期非目标

以下内容不应塞进一期：

- 日志本重命名
- 新增日志本
- 切换日志本 UI
- 单条记录原地编辑
- 单条记录删除
- 分页明细页
- 复杂筛选器
- 多渠道同步 UI

原因很简单：

- 这些能力不是现有仓库里“随手一接就能用”的成熟 service
- 一上来就做，会把首版从“交互增强”拖成“数据层改造”

## 5. 入口策略

### 推荐入口

稳定入口建议优先用：

- `工作日志`
- 现有关键词族中的文本命令

### 可选新增入口

以下入口可以做，但属于新增别名：

- `/worklog`

注意：

- `/worklog` 不能在文档里默认写成“当前已经存在的命令”
- 它只是推荐新增入口

## 6. 交互原则

- 按钮优先：高频操作尽量一眼能点
- 文本兼容：老命令不废
- 页面简短：一屏只表达一个意图
- 路径可回退：任何时候都能返回主菜单或取消
- 业务不重复：按钮层只做 UI 路由与状态，不复制业务逻辑

## 7. 建议页面

### 7.1 主菜单卡片

标题：

**工作日志**

正文建议：

请选择操作：

按钮：

- `➕ 记录日志` | `📋 今日记录`
- `📊 本月统计` | `⚙️ 帮助`

说明：

- 一期先别把“日志管理”塞进主菜单
- 首版入口要尽量少，避免把低频管理项顶到首页

### 7.2 记录日志卡片 / 输入态

标题：

**记录工作日志**

正文建议：

请直接发送内容。
示例：
`工作项：修复发布脚本回显`
`工时：2`

按钮：

- `📝 直接输入内容`
- `⏱ 只填工时`
- `❌ 取消`

关键点：

- 点击按钮后进入输入态，而不是直接写入
- 输入态必须按 `channel + sender_id` 隔离
- 若在群聊里，还应纳入会话上下文，避免串话

### 7.3 今日记录卡片

展示建议：

- 今日记录条数
- 今日总工时
- 最近一条摘要

按钮：

- `➕ 再记一条`
- `📄 查看全部（可留作后续）`
- `⬅️ 返回`

注意：

- 一期允许“查看全部”先不做，按钮可以不放
- 空状态必须有明确文案和返回入口

### 7.4 本月统计卡片

展示建议：

- 本月累计工时
- 应完成工时
- 未完成工时

按钮：

- `⬅️ 返回`
- `🧾 今日锐评` 仅在现有锐评入口已经可复用时再放

注意：

- 文档里不要把“查看明细”“生成锐评”默认当成现成能力
- 这些必须建立在已有 service 可调用的前提下

## 8. 文本命令共存策略

当前文档里最容易写虚的一点，就是把“计划新增别名”和“当前已有命令”混为一谈。

正确口径应该是：

### 现有命令能力

优先兼容：

- `工作日志=`
- `工作日志列表`
- `当前工作日志`
- `记工作日志：`
- `记工作日志@`
- `工作日志口令：`

### 建议新增别名

可新增：

- `工作日志`
- `/worklog`

### 原则

- 按钮流程不能绕开现有命令/service 的权限和写入约束
- 按钮只是多一个入口，不是多一套后端

## 9. 状态机建议

建议最小状态集：

- `idle`
- `awaiting_worklog_input`
- `awaiting_hours_only`

一期先到这里即可。

以下状态不要在一期文档里写成“默认会做”：

- `awaiting_logbook_rename`
- `awaiting_logbook_create`
- `awaiting_edit_content`
- `awaiting_edit_hours`
- `awaiting_delete_confirm`

这些都属于二期以后。

### 状态字段建议

- `senderKey`：建议包含 channel + sender_id
- `state`
- `updatedAt`
- `expiresAt`
- `context`
  - `entryMode`
  - `requestedBook`
  - `sourceMessageId`（如需要）

## 10. callback 设计原则

- Telegram `callback_data` 长度上限很紧，必须走紧凑编码
- 不要在 callback 里直接塞长文本、文件路径、长 UUID
- 对于需要定位具体记录的场景，优先传短 token，再由服务端映射
- 一期的 callback 应尽量固定枚举，减少动态参数

## 11. 技术边界

建议实现层次：

- UI 层：按钮文案、卡片内容、callback 路由
- 状态层：输入态与超时回收
- 服务层：复用现有 `config / state-store / access / guards / worklog-storage`

不要做的事：

- 不要复制 append / locate / access 逻辑
- 不要把 Telegram 流程写成独立数据模型
- 不要一边卡片一边文本，各写一套解析器

## 12. 验收定义

一期完成的标准应改成这 4 条：

- 可以进入主菜单
- 可以从按钮进入记录日志输入态并成功写入
- 可以查看今日记录摘要
- 可以查看本月统计摘要

只要这 4 条成立，且原有文本命令不回归，就足够进入验证。

一句话结论：

先把“能点、能记、能看、能查”做实，别一上来就把编辑、删除、日志本管理全写进首版承诺里。
