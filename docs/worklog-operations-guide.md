# Worklog Agent 操作手册

这份文档面向实际执行任务的 agent，不是设计说明。

适用对象：

- 通过 Telegram / 其他消息渠道代用户记录日志的 agent
- 日常使用 `/worklog` 的普通用户
- 维护 sender 绑定、日志本、Web 预览的管理员

## 1. 先读规则文件

在 agent 代用户记录日志前，**先看规则文件**：

- `config/worklog-writing-rules.md`

这份规则文件是当前插件的统一写作规范，至少包含：

- 每日工作项在展示、复制、导出时必须带序号
- 录入时不要保留手工序号，系统会统一编号
- 润色只能做客观、简洁、可复盘的改写
- 锐评应先判断“值不值得写”，再决定是否建议补写

如果用户只是口头描述工作内容，agent 应先按规则整理，再写入。

## 2. Agent 的默认执行顺序

推荐遵循下面的顺序：

1. 识别用户是要“记工作项”还是“补锐评”
2. 先按 `config/worklog-writing-rules.md` 整理内容
3. 如果是自然语言工作项，优先走“草稿确认”
4. 如果启用了 AI，可先做 AI 润色 / AI 锐评检测
5. 最终确认后再落盘
6. 写入后返回 `/worklog today` 或成功卡片，便于用户复核

## 3. 工作项记录规则

### 3.1 编号规则

- 每天的工作项最终都按当天顺序显示为：`1. xxx`、`2. xxx`
- 用户或 agent 输入时，不要手工固定编号
- 即使输入里带了 `1. `、`2. `，写入前也会去掉，再由系统重新编号

### 3.2 润色规则

agent 整理工作项时，应满足：

- 保留客观事实，不虚构结果
- 一条记录只描述一个主要事项
- 优先使用可复盘动作词：排查、修复、联调、补齐、验证、发布、回滚、跟进
- 避免空话：如“处理了一些事情”“推进相关工作”
- 不要写成周报或夸功文本

### 3.3 重复规则

- 同一天内若工作项归一化后重复，插件会按 `skipped` 处理，避免重复记账
- 所以 agent 不要为了“凑完整”重复写同一个事项

## 4. 锐评规则

### 4.1 什么时候适合补锐评

优先记录：

- 阻塞
- 取舍
- 风险
- 返工原因
- 协作问题
- 后续建议
- 经验总结

如果当天只是常规执行、没有明显判断价值，可以不补锐评。

### 4.2 写法要求

- 简洁
- 具体
- 可复盘
- 不喊口号
- 不空泛总结
- 不把锐评混写进工作项

### 4.3 当天锐评限制

默认：

- `commentPolicy.allowSameDayComment = false`

也就是：

- 默认不建议给“今天”即时补锐评
- 更适合给历史日期补锐评
- 若管理员显式改成 `true`，才允许当天补写

## 5. 普通用户常用命令

### 5.1 直接写入

- `/worklog 1.5 修复筛选回显`
- `/worklog append 2 联调 Telegram 卡片`

适合：

- 内容已经足够明确
- 不需要先走草稿确认

### 5.2 自然语言写入

- `/worklog 今天联调 Telegram 卡片 2 小时`
- `/worklog 昨天修复 sender 绑定 1.5 小时`
- `/worklog 工作项：联调 Telegram 卡片，工时：2`

当前行为：

- 先生成待确认草稿
- 用户点“写入”后才真正落盘
- 用户点“修改”后，可重新发送新的 `/worklog ...` 覆盖草稿

### 5.3 先选工时，再补内容

- `/worklog ah`
- `/worklog item 修复筛选回显`

适合：

- 用户已经先决定工时
- 但工作项文本还没整理好

### 5.4 查询

- `/worklog today`
- `/worklog month`
- `/worklog recent`
- `/worklog web`

### 5.5 编辑 / 删除

- `/worklog edit 2026-03-08 1`
- `/worklog replace 1.5 新内容`
- `/worklog delete 2026-03-08 1`

说明：

- Telegram 的今日卡片前 5 条有 `✏️ / 🗑` 快捷按钮
- 删除是两步确认，不会直接一键删除

## 6. 锐评与 AI 相关命令

### 6.1 手动补锐评

- `/worklog comment 2026-03-06 这天主要是在清理遗留问题。`
- `/worklog comment 2026-03-06`

说明：

- 带内容时：直接补或覆盖当天锐评
- 只带日期时：打开“补锐评 / 改锐评”提示卡片

### 6.2 AI 润色工作项

- `/worklog ai-polish`

说明：

- 只对“当前待确认工作项草稿”生效
- 会按规则文件先润色工作项
- 不会自动写入，仍需用户确认

### 6.3 AI 检测锐评

- `/worklog ai-comment 2026-03-06`
- `/worklog ai-comment`

说明：

- AI 先判断当天是否值得补锐评
- 若不值得写，会直接返回“不建议补锐评”及原因
- 若值得写，会生成一条待确认锐评草稿
- 最终仍需用户确认保存

## 7. Telegram 卡片交互规则

### 7.1 当前特点

- Telegram 下优先复用同一张卡片
- 尽量原地更新，不刷满聊天窗口
- 卡片有有效期，过期后重新发 `/worklog`

### 7.2 记录日志卡片

可能出现的按钮：

- `✅ 写入`
- `✏️ 修改`
- `🪄 AI 润色`
- `❌ 取消`
- `⬅️ 主菜单`

### 7.3 今日记录卡片

可能出现的按钮：

- `➕ 再记一条`
- `✏️ 1 / 2 / 3 ...`
- `🗑 1 / 2 / 3 ...`
- `💬 补锐评 / ✏️ 改锐评`
- `🧠 检测锐评`

说明：

- 若配置禁止当天补锐评，则今天页面不会出现“补锐评”按钮
- 历史日期仍可通过命令显式补锐评

## 8. 管理员命令

### 8.1 日志本管理

- `/worklog books`
- `/worklog create <key> <名称>`
- `/worklog rename <key> <新名称>`

### 8.2 sender 绑定管理

- `/worklog bb`
- `/worklog bind <sender> <book>`
- `/worklog unbind <sender>`
- `/worklog bindings [页码] [关键字]`

说明：

- 这些改的是运行时状态
- 会写入插件状态文件里的 `senderBindings`
- 不会直接改宿主主配置里的静态绑定

### 8.3 全局当前日志本切换

仅当 `senderRouting.mode = current` 时可用：

- `/worklog use <book>`

若当前是 `by_sender_id`：

- 不应切全局当前日志本
- 应通过 sender 绑定解决

## 9. Web 预览

### 9.1 获取地址

- `/worklog web`

### 9.2 当前行为

- 返回可访问地址，不再回 `0.0.0.0`
- 若配置了公网入口 / 反代，则优先返回公网入口
- 非管理员读取仍受口令保护

### 9.3 常见结构

类似：

- `http://<host>:3210/worklog-preview?senderId=telegram%3A6684352915&month=2026-03&book=u-telegram-6684352915`

## 10. 安全边界

### 10.1 agent 可默认直接做的事

- 当前对话内查询
- 普通日志追加
- 单条编辑
- 删除确认后的单条删除
- 手动补锐评
- 运行时 sender 绑定调整
- 运行时日志本创建 / 重命名

### 10.2 agent 不应默认静默做的事

- 改宿主主配置
- 删除非空日志本目录
- 删除静态配置里的日志本
- 越过 `restrictedPathPrefix` 的目录写入
- 在未确认前直接替用户保存 AI 生成的锐评

### 10.3 原因

因为 `worklog` 同时承担：

- sender 路由
- 权限校验
- 文件落盘
- 在线预览
- Telegram 卡片交互

高风险动作必须保留确认流，不能默认“顺手就改”。

## 11. 常见故障排查

### 11.1 `sender-not-allowed-for-auto-bind`

优先检查：

- `senderRouting.allowAutoBindSenders`
- `senderRouting.bindings`
- 运行时 `senderBindings`
- 调用方是否传错 senderId

### 11.2 Telegram 没按钮 / 只剩文本

优先检查：

- 插件是否已重新编译
- 网关是否已重启
- 卡片状态是否过期或损坏
- Telegram `inlineButtons` 能力是否开启

### 11.3 `/worklog today` 显示今天没记录

通常不是整本没日志，而是：

- 当月文件存在
- 但当天 `YYYY-MM-DD` 分节还没写入

### 11.4 AI 命令不可用

优先检查：

- `ai.enabled` 是否已开启
- `ai.model` 是否已配置
- `ai.apiKeyEnv` 指向的环境变量是否存在
- `ai.baseUrl` 是否为兼容 `chat/completions` 的接口

### 11.5 Web 地址打不开

优先检查：

- `preview.host / preview.port / preview.basePath`
- 网关进程是否监听该端口
- 反向代理 / 防火墙
- 浏览器访问时是否已完成读权限授权

## 12. 推荐操作流

### 12.1 用户日常使用

1. `/worklog`
2. 记录日志或直接输入自然语言
3. 需要时点 `AI 润色`
4. 确认写入
5. `/worklog today` 复核结果

### 12.2 agent 代用户记录

1. 先读 `config/worklog-writing-rules.md`
2. 判断是工作项还是锐评
3. 整理成符合规则的内容
4. 必要时走 AI 辅助
5. 落盘后返回今日记录或成功卡片

### 12.3 管理员维护

1. `/worklog books`
2. 必要时执行：
   - `create`
   - `rename`
   - `bind`
   - `unbind`
3. 用 `/worklog bindings` 复核运行时绑定
4. 用 `/worklog web` 验证预览入口
