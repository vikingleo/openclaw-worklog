# Worklog 管理员危险操作清单

这份清单只面向管理员。

目标不是教你“怎么快点删”，而是避免：

- 误删日志本
- 误解绑 sender
- 把静态配置和运行时状态搅在一起
- 让 agent 在高风险动作上默认乱来

## 1. 删除前先判定对象类型

先分清楚你要处理的是哪种日志本：

- 静态配置日志本：来自插件配置里的 `books`
- 运行时日志本：来自状态文件里的 `state.books`

默认原则：

- 静态配置日志本不要在卡片里直接删
- 运行时日志本可以走归档 / 删除流程

## 2. 非空目录先归档，不直接删除

删除前先检查目录是否为空。

- 空目录：才允许走“删除空日志本”
- 非空目录：先归档，再人工确认后处理

推荐顺序：

1. `/worklog ba <book>`
2. `/worklog baa <book>`
3. 确认归档目录存在
4. 再决定是否做后续人工清理

## 3. sender 绑定调整后必须复核

执行这些命令后：

- `/worklog bind <sender> <book>`
- `/worklog unbind <sender>`

应立即复核：

- `/worklog bindings`
- `/worklog bindings 1 <sender关键字>`

不要凭记忆认为“已经绑好了”。

## 4. 不要把运行时绑定误当主配置

当前 `bind / unbind` 管理的是：

- 运行时 `senderBindings`

它不是：

- 宿主主配置里的静态 `senderRouting.bindings`

所以：

- 临时运营调度，用运行时绑定
- 长期固定关系，再回主配置收口

## 5. 归档会影响哪些状态

归档运行时日志本时，会同步移除：

- 活动日志本列表中的该书本
- 指向该书本的运行时 sender 绑定
- 若它恰好是运行时 `currentBook`，也会一起解除

因此归档后要复核：

- `/worklog books`
- `/worklog bindings`

## 6. 删除动作的最低确认标准

满足下面 4 条再删：

- 该日志本是运行时创建的
- 目录为空
- 没有残留 sender 绑定
- 你确认不是当前仍在使用的活动日志本

如果其中任何一条拿不准：

- 不删
- 先归档

## 7. Web 与日志本管理不要混着改

在做以下动作时，不建议同时改 Web / 反代 / 预览 host：

- 绑定迁移
- 日志本归档
- 日志本删除

原因：

- 出问题时很难分清是数据层、权限层还是访问层的问题

正确做法：

- 先稳定日志本状态
- 再处理 Web 访问问题

## 8. 给 agent 的执行边界

建议保持这条原则：

可以默认直接做：

- 查询
- 追加日志
- 单条编辑
- 单条删除确认后执行
- 运行时绑定调整
- 运行时日志本创建 / 重命名

不应默认静默做：

- 修改宿主主配置
- 删除非空日志本目录
- 删除静态配置日志本
- 对外服务监听方式变更
- 超出受控目录前缀的文件写入

## 9. 推荐实操口令顺序

### 调整绑定

1. `/worklog bindings 1 <关键字>`
2. `/worklog bind <sender> <book>`
3. `/worklog bindings 1 <关键字>`

### 清理临时日志本

1. `/worklog books`
2. `/worklog ba <book>`
3. `/worklog baa <book>`
4. `/worklog books`
5. `/worklog bindings`

### 删除空日志本

1. `/worklog books`
2. 确认目录为空
3. `/worklog bd <book>`
4. `/worklog bdd <book>`

## 10. 结论

管理员最容易犯的错，不是“不会操作”，而是：

- 以为自己很确定
- 于是跳过确认
- 最后把运行时状态、静态配置、目录数据三件事一起改乱

所以真正的铁律只有一句：

- **不确定就先查；要清理就先归档；删之前一定复核。**
